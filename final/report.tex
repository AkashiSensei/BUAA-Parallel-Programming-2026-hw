\section{NVIDIA GPU体系结构与CUDA编程模型}

在进入 GPU 矩阵相乘实验之前，我们首先需要对NVIDIA GPU体系结构以及CUDA编程模型建立基本的了解。

\subsection{NVIDIA GPU硬件结构}

NVIDIA GPU硬件包含以下几个主要结构：GPU处理集群（GPU Processing Clusters，GPC）、纹理处理集群（Texture Processing Clusters，TPC）、流式多处理器（Streaming Multiprocessors，SM）和CUDA核心（Streaming Processor，SP）。

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{plots/nvidia_cuda_ga100_overview.png}
    \caption{NVIDIA GA100核心全景图}
    \label{fig:nvidia-ga100-overview}
\end{figure}

图~\ref{fig:nvidia-ga100-overview}（NVIDIA，2020）展示了Ampere架构GPU核心GA100的结构。GPC是NVIDIA GPU架构中最高级别的结构单元，负责处理大量并行任务，实现高吞吐量的计算能力，其中包含若干TPC和其他一些共享资源，例如L2缓存。TPC是次一级的集群，主要负责处理纹理相关的计算任务。它包含若干SM，后者负责执行具体的CUDA线程块，其结构如图~\ref{fig:nvidia-ga100-sm}所示，包含四个SMSPs（SM sub partitions），是SM上的主要处理单元，管理一个固定大小的warp池。而SMSPs具有若干CUDA核心、调度器（scheduler）、寄存器堆、共享内存和其他辅助硬件。CUDA核心是最基本的计算单元，执行CUDA程序中的线程指令，分为不同类型，包括执行浮点运算的FP32和FP64核心、执行整数运算的INT32核心，以及进行深度学习加速运算的张量计算核心（Tensor Cores）。每个SM包含的CUDA核心数量通常较多，这使得GPU能够高效地执行并行计算任务。当计算一款GPU的CUDA核心总数时，通常统计其FP32核心的总数，因为FP32运算是GPU中最为常见的运算类型。

\begin{figure}[H]
    \centering
    \includegraphics[width=0.72\textwidth]{plots/nvidia_cuda_ga100_sm.png}
    \caption{GA100核心SM结构图}
    \label{fig:nvidia-ga100-sm}
\end{figure}

\subsection{CUDA编程模型}

本小节将对CUDA编程模型中的kernel核函数、线程层次结构、存储层次结构以及同步操作进行简要介绍。

在CUDA编程模型中，kernel核函数是在GPU设备上执行的并行计算单元，其核心设计思想是将计算任务分解为大量可同时执行的线程。与CPU上的顺序执行函数不同，kernel核函数在调用时需要指明线程组织方式，表明执行该核函数所需的线程数量和结构。每个线程独立执行核函数逻辑，通过线程索引访问输入数据或控制计算行为，从而实现数据并行或任务并行计算。

在计算能力9.0之前，CUDA的线程层次结构采用三级组织方式：最外层为线程网格（Grid），中间层为线程块（Block），最内层为线程（Thread），如图~\ref{fig:nvidia-cuda-thread-hierarchy}（a）（NVIDIA，2023）所示。Grid由Block组成，Block由Thread组成，其形状由\texttt{dim3}类型的参数指定，支持一维、二维或三维布局，但其大小受具体GPU计算能力的限制。而在计算能力9.0中，NVIDIA在Block和Grid之间引入了一个新的层级线程块集群（Thread Block Cluster），如图~\ref{fig:nvidia-cuda-thread-hierarchy}（b）（NVIDIA，2023）所示。

\begin{figure}[H]
    \centering
    \begin{subfigure}{0.48\textwidth}
        \centering
        \includegraphics[width=\linewidth]{plots/nvidia_cuda_thread_hierarchy_ampere.png}
        \caption{Ampere架构线程层次结构图}
        \label{fig:nvidia-cuda-thread-hierarchy-ampere}
    \end{subfigure}
    \hspace*{\fill}
    \begin{subfigure}{0.48\textwidth}
        \centering
        \includegraphics[width=\linewidth]{plots/nvidia_cuda_thread_hierarchy_hopper.png}
        \caption{Hopper架构线程层次结构图}
        \label{fig:nvidia-cuda-thread-hierarchy-hopper}
    \end{subfigure}
    \caption{CUDA线程层次结构图}
    \label{fig:nvidia-cuda-thread-hierarchy}
\end{figure}

在执行时，Block Cluster中所有线程共同调度到一个GPC上，而Block中所有线程将会以32个为一组划分到不同warp，被调度到同一个SM中。warp是NVIDIA GPU调度的基本单位，其中所有线程以SIMD（单指令流多数据流）方式执行相同指令。

\begin{figure}[H]
    \centering
    \includegraphics[width=0.82\textwidth]{plots/nvidia_cuda_memory_hierarchy.png}
    \caption{CUDA存储层次结构图}
    \label{fig:nvidia-cuda-memory-hierarchy}
\end{figure}

存储层次结构是CUDA编程模型的另一核心要素，其设计旨在平衡访存速度与存储容量和可见范围，如图~\ref{fig:nvidia-cuda-memory-hierarchy}（NVIDIA，2025）所示。其中容量最大的为全局内存（Global Memory），能够被所有核函数访问，但其延迟较高。\texttt{cudaMemcpy()}系列函数可以在主机和设备的全局内存间进行数据拷贝。同样被所有核函数共用的还包括纹理内存（Texture Memory）与常量内存（Constant Memory），这二者均为只读缓存型内存，前者针对二维数据寻址优化，支持硬件级数据插值；后者适用于核函数中频繁访问的常量数据，通过缓存机制减少全局内存访问次数。而共享内存（Shared Memory）则是隶属于线程块的高速存储单元，容量和访问延迟均显著小于全局内存，位于SM内部，其作用域限于当前线程块内的所有线程。寄存器（Register）是每个线程独占的最快存储单元，用于存储线程私有临时变量，寄存器的分配由编译器自动管理。此外，在计算能力9.0以上的架构中，在全局内存和共享内存之间，还有一层分布式共享内存（Distributed Shared Memory），比全局内存更快，作用域限于同一线程块集群中的线程。

最后，CUDA提供了多个同步函数，用于控制核函数的执行顺序和数据一致性。\texttt{cudaDeviceSynchronize()}函数用于在主机代码中同步设备操作，确保之前启动的所有核函数在继续执行主机代码之前完成；而\texttt{\_\_syncthreads()}则是一个内置的线程块级别的同步函数，在核函数中使用，用于在线程块内同步所有线程，每个线程在继续执行之前，都会等待所有线程都到达同步点。

此外，在CUDA程序设计和性能分析中，Stream和Context的概念也同样重要。

Stream是CUDA中用于管理异步操作的关键机制。它就像是一条任务流水线，在这个流水线里可以有序地安排各种CUDA操作，像核函数的调用、内存的拷贝等。每个Stream内的操作是按顺序依次异步执行的，不同Stream之间的操作可以并行开展。

Context可以看作是CUDA设备的一个独立运行环境。它包含了CUDA程序运行所需的各种状态信息，如内存分配情况、内核函数的编译状态等，每个Context都有自己独立的资源和状态。Context就像是一个容器，把不同的CUDA任务隔离开来。

Stream和Context，在某些方面类似于CPU程序设计概念中的线程和进程。






\section{实验内容与方法}

本实验在实验节点裸机上完成 CUDA 程序的开发、运行与性能分析，不依赖容器环境。实验采用渐进式路线：先实现仅使用全局内存的朴素 GPU 矩阵乘法 kernel，借助 Nsight Compute（\texttt{ncu}）观察其性能特征与瓶颈；再在此基础上引入共享内存分块等优化，复测并对比分析。除自研实现外，使用 cuBLAS 生成标准结果并作为性能对照上界。各阶段均使用预先准备的固定测试矩阵，以保证结果可复现、计时可比。

\subsection{计算问题与数据类型}

实验求解双精度方阵乘法
\begin{equation}
  C = A \times B,\quad A,B,C \in \mathbb{R}^{N \times N}.
\end{equation}
矩阵在主机端与设备端均按\textbf{行主序}（row-major）连续存储。自研 kernel 与 cuBLAS 调用采用相同数据类型（\texttt{double}），以便正确性比对与性能对比具有可比性。

对稠密矩阵乘法而言，在相同 $N$ 与数据类型下，矩阵元素的具体数值不改变运算规模与访存模式，因此\textbf{随机矩阵的取值不影响 GPU 计算性能}；实验只需固定随机种子以保证可复现。本实验不涉及稀疏矩阵存储或稀疏 kernel，矩阵中零元素的比例不作为实验变量。

\subsection{测试数据准备}

为避免在每次性能测试时重复进行耗时的主机端矩阵乘法，实验在\textbf{数据准备阶段}一次性完成以下工作：

\begin{enumerate}
  \item 给定固定随机种子，为若干组矩阵规模 $N$ 分别生成 $A$、$B$，并以二进制格式落盘保存；
  \item 调用 cuBLAS（\texttt{cublasDgemm}）在 GPU 上计算 $C_{\mathrm{ref}} = A \times B$，将参考结果同样落盘保存；
  \item 后续 benchmark 仅读取已保存的 $A$、$B$（及校验所需的 $C_{\mathrm{ref}}$），不再重新生成数据或重复计算参考结果。
\end{enumerate}

计划覆盖的矩阵规模如表~\ref{tab:matmul-sizes} 所示（可在实现阶段根据显存与耗时微调）。各 $N$ 对应一组固定的 $(A,B,C_{\mathrm{ref}})$，所有 kernel 与 cuBLAS 均复用同一组输入。

\subsection{自研 kernel 设计}

自研实现采用\textbf{一个 kernel 覆盖多种矩阵规模}的方式：在主机代码中根据 $N$ 计算网格与线程块维度（含对不整除 tile 的边界处理），再启动相应 kernel，而非为每个 $N$ 单独编写核函数。

\subsubsection{朴素全局内存 kernel（阶段一）}

第一阶段实现不使用共享内存的 naive kernel。每个线程负责计算 $C$ 中一个元素 $C_{row,col}$，内层对 $k=0,\ldots,N-1$ 累加 $A_{row,k}\cdot B_{k,col}$，所需数据均直接从全局内存读取。

该 kernel 的主要目的在于：（1）完成 CUDA 内存分配、数据传输与 kernel 启动的基本流程；（2）建立后续优化前的性能 baseline；（3）通过 \texttt{ncu} 观察典型 memory-bound 行为（如全局访存过多、合并度不足、内存延迟导致的 stall 等）。线程块大小在实现阶段选取常用配置（如 $16\times 16$），并在各 $N$ 下保持一致以便对比。

\subsubsection{共享内存分块 kernel（阶段二）}

第二阶段在 naive 实现基础上引入\textbf{共享内存（Shared Memory）分块}优化。将 $A$、$B$ 的子块载入线程块内的共享内存，在 tile 内复用数据，减少对全局内存的重复访问；块内使用 \texttt{\_\_syncthreads()} 保证协作加载与计算的正确顺序。对 $N$ 不能整除 tile 大小的边界情形，在 kernel 内通过条件判断处理部分 tile。

tile 宽度（如 $16$ 或 $32$）作为编译期常量或可调参数，在实验中对各 $N$ 采用统一配置；若后续 profiling 显示 occupancy 或 bank conflict 存在问题，可在报告中讨论 padding 等进一步改进方向，但不作为本实验的必需步骤。

\subsection{cuBLAS 对照实验}

除自研 kernel 外，实验调用 cuBLAS 中的 \texttt{cublasDgemm} 完成同一组 $(A,B)$ 上的矩阵乘法，承担两类角色：

\begin{itemize}
  \item \textbf{标准结果}：数据准备阶段生成的 $C_{\mathrm{ref}}$ 作为 gold standard，用于校验 naive 与 tiled kernel 的数值正确性；
  \item \textbf{性能上界参照}：在同一硬件与数据规模下，将 cuBLAS 的 GPU 计算耗时作为工业级优化实现的参照，衡量自研 kernel 与库函数之间的差距。
\end{itemize}

调用 cuBLAS 时需注意其默认列主序约定：通过合理设置转置标志或等价变换，使 \texttt{cublasDgemm} 与自研 kernel 在\textbf{相同的行主序布局与 leading dimension} 下计算同一问题，避免因存储约定不同导致结果或性能不可比。

\subsection{正确性验证方法}

对每个规模 $N$ 与每种实现，在性能计时之外单独进行正确性检查：将 GPU 计算得到的 $C$ 与已保存的 $C_{\mathrm{ref}}$ 逐元素比较。由于浮点累加顺序不同，允许存在舍入误差，采用最大相对误差等指标判定：
\begin{equation}
  \mathrm{err}_{\mathrm{rel}} = \max_{i,j}\frac{|C_{ij}-C_{\mathrm{ref},ij}|}{|C_{\mathrm{ref},ij}|+\varepsilon},
\end{equation}
其中 $\varepsilon$ 为极小正数以防除零。仅当误差低于预设阈值时，认为该实现通过验证。正确性校验不计入 kernel 性能计时段。

\subsection{性能测试方法}

性能测试在单块 GPU 上独立进行（如设置 \texttt{CUDA\_VISIBLE\_DEVICES=0}），避免多用户争用干扰计时。采用 CUDA 事件（\texttt{cudaEvent}）测量\textbf{GPU 端矩阵乘法核函数执行时间}（含 cuBLAS 调用中与计算直接相关的部分）；主机端数据生成、文件读写与结果校验均不计入计时。

具体流程如下：

\begin{enumerate}
  \item 将 $A$、$B$ 拷贝至设备端（若实验设计为“数据已在 GPU”，则可在预热阶段完成拷贝，计时仅包围 kernel 或 \texttt{cublasDgemm} 本体，并在报告中明确说明）；
  \item 执行若干次 warmup，使 GPU 进入稳定状态；
  \item 重复运行多次，取平均时间作为该 $(N,\ \text{实现})$ 的最终耗时；
  \item 对 naive kernel、tiled kernel 与 cuBLAS 在\textbf{相同 $N$ 与相同输入}下分别测时，计算 tiled 相对 naive 的加速比，以及自研实现相对 cuBLAS 的性能比。
\end{enumerate}

\subsection{性能分析方法}

使用 Nsight Compute（\texttt{ncu}）对代表性规模（如 $N=1024$ 与 $N=4096$）分别 profile 朴素 kernel 与共享内存 kernel，重点观察并对比以下指标：

\begin{itemize}
  \item 全局内存吞吐量（Memory Throughput）与访存效率；
  \item 计算吞吐量（Compute Throughput）与 SM 占用率（Achieved Occupancy）；
  \item warp stall 原因（如 Memory Dependency、Long Scoreboard 等）；
  \item 共享内存优化后全局 load/store 是否减少，occupancy 是否因 shared memory 用量发生变化。
\end{itemize}

通过``实现优化前 $\rightarrow$ 实现优化后''的对比，结合 \texttt{ncu} 报告解释 shared memory 分块为何能缓解 naive kernel 中的 memory-bound 瓶颈；性能曲线与 profiling 结果留待第~\ref{sec:exp-result} 节给出。

\subsection{标准结果生成（cuBLAS）}

\subsubsection{程序流程}

在输入矩阵就绪后，使用 \texttt{gen\_cref.cu} 读取 \texttt{N\{N\}.bin}，将 $A$、$B$ 拷贝至 GPU，调用 cuBLAS 的 \texttt{cublasDgemm} 计算 $C_{\mathrm{ref}}=A\times B$，再将结果写回主机并保存为 \texttt{N\{N\}\_cref.bin}。该步骤在数据准备阶段\textbf{一次性}完成；后续正确性校验直接加载已保存的 $C_{\mathrm{ref}}$，不计入 kernel 性能测试时间。

由于 cuBLAS 按列主序（column-major）接口约定存储矩阵，而本实验输入/输出均为行主序，程序采用等价变换
\begin{equation}
  (AB)^{\mathsf T} = B^{\mathsf T}A^{\mathsf T},
\end{equation}
在调用 \texttt{cublasDgemm} 时对 $A$、$B$ 使用转置操作，使 GPU 上列主序运算的结果与行主序意义下的 $C=AB$ 一致。计算在 \texttt{dell-65} 的单块 GPU（\texttt{CUDA\_VISIBLE\_DEVICES=0}）上执行。
